## 一、恒星与轨道修正

**问题**：恒星亮度过大；神谕之境、世末之城离恒星太近。

**改动**（`planets/outing_sun.json`）：

| 参数 | 原值 | 新值 | 说明 |
|------|------|------|------|
| `solarLuminosity` | 3 | **1.0** | 光照强度降到原来的 1/3，这是亮度的主控杆 |
| `Star.size` | 2000 | **1500** | 缩小恒星视觉半径，避免占满天空 |
| `Base.surfaceSize` | 2000 | **1500** | 与 size 同步 |
| `lightRadius` | 50000 | 60000 | 光照范围略增，保证推远后的星球仍被照到 |

**轨道推远**：

| 星球 | `semiMajorAxis` 原值 | 新值 | 距恒星表面间距（半径 1500）|
|------|------|------|------|
| 神谕之境 (`god_realm.json`) | 3000 | **7000** | 约 5500（原来仅 1000，几乎贴着恒星）|
| 世末之城 (`singer_world.json`) | 8000 | **15000** | 约 13500 |

> 神谕之境仍比世末之城更靠近恒星，符合"靠近恒星运行的小星球"的设定。
> 世末之城本身是"末日昏暗"基调（`atmosphereSunIntensity:0.6` + `ambientLight:1.2`），即使恒星变暗也不会全黑。

## 三、文本接线与 Ship Log 解锁

**之前的问题**：`singer_diary.xml`、`bet_stele.xml` 内容写得很好，但**没有任何星球 JSON 引用它们**，游戏里根本看不到；而且 Ship Log 的探索事实几乎没有任何解锁触发器。

### 3.1 用 translatorText 接入文本

- `singer_world.json` → 广场墙壁放歌者日记（`type:"wall"`, `seed:42`）。
- `god_realm.json` → 放赌约石碑（`type:"wall"`, `seed:7`）+ 一个"录音石"占位 detail。

### 3.2 文本读取解锁日志（ShipLogConditions）

- `singer_diary.xml`：读到第 1 段 → 解锁 `SINGER_DIARY_FACT`。
- `bet_stele.xml`：读完 1,2 段 → `STELE_EXPLORE`；读完 1,2,3 段 → `THIRD_PARTY_FACT` + `THIRD_PARTY_LINK`。

### 3.3 revealVolumes（走到即解锁）

**世末之城**（`Volumes.revealVolumes`）：
- 着陆/广场区(r=25)：`SINGER_CITY_EXPLORE` `SQUARE_EXPLORE` `SQUARE_RUMOR` `TOWER_RUMOR`
- 音乐厅(r=18)：`HALL_EXPLORE` `AMPLIFIER_FACT`
- 高塔(r=18)：`TOWER_EXPLORE` `TOWER_PART_FACT` `GOD_REALM_RUMOR`

**神谕之境**：
- 着陆区(r=40)：`GOD_REALM_EXPLORE` `STELE_RUMOR`
- 录音石(r=8)：`GOD_VOICE_FACT`

### 3.4 signal 识别解锁

歌者信号"歌者之声"加 `"reveals": "HALL_RUMOR"`，并把信号源从广场中心移到音乐厅舞台 (0,302,-58)，`detectionRadius` 800→1000，让玩家循着歌声找到音乐厅。

### 3.5 好奇心配色

`outing_system.json` 补 `curiosities` 定义 `OUTING_MYSTERY` 为暗紫色（呼应"鸥停之谜"）。

### 3.6 新增的两个 Fact

`singer_city_log.xml` 补 `AMPLIFIER_FIXED`（扩音器修复）和 `TRUE_ENDING_FACT`（结局），供对话和结局判定揭示。

## 四、对话与扩音器修复

### 4.1 天依对话（`tianyi_dialogue.xml`）

新增"修复后"分支 `AfterRepair`（`EntryCondition: AMPLIFIER_REPAIRED`），并**放在文件最前**——OW 对话按顺序取第一个满足 EntryCondition 的节点，DEFAULT 永远满足，所以条件分支必须排在 DEFAULT 之前。

- 未修复：呆望天空，"……下雨了。一直在下。"
- 已修复：听到歌声，迈出一步又停下，"我应该……过去看看吗？"（设 `TIANYI_HEARD_SONG`）

### 4.2 扩音装置交互（新建 `amplifier_dialogue.xml`）

放在音乐厅舞台后方 (3,301,-62)。交互 → 选择"嵌入零件修复" → 设置 `AMPLIFIER_REPAIRED`（本循环条件）+ `AMPLIFIER_EVER_REPAIRED`（持久）+ 揭示 `AMPLIFIER_FIXED`。已修复后再交互只显示运转提示。

> ⚠ **本轮简化**：原设计的"找到零件 A(广场)+零件 B(高塔)→插入插槽"完整 fetch-quest，本轮**简化为一次对话交互即可修复**。
> 原因：NH 的 `item`/`itemSocket` schema 我没有十足把握，写错可能导致整颗星球加载失败（反而让星球变荒凉）。
> 当前用纯对话系统实现，是我**有把握能正确加载**的方案，叙事主线照样跑通。
> 完整的零件拾取/插槽谜题列为后续增强（见第七节）。

## 五、结局判定（`EndingJudge.cs` + 主 cs 接入）

### 5.1 EndingJudge.cs

之前是空文件，现在实现为一个 MonoBehaviour：
- 每 0.5 秒轮询 `DialogueConditionManager.SharedInstance.GetConditionState("AMPLIFIER_REPAIRED")`。
- 条件为真时触发一次 `TriggerTrueEnding()`：
  - `Locator.GetShipLogManager().RevealFact("TRUE_ENDING_FACT")` 揭示结局日志
  - `NotificationManager` 弹出屏幕通知："雨停了。云层裂开，阳光穿透乌云。轮回终结，世界得赎。"
- 用了 OW 内置的稳定 API（DialogueConditionManager / Locator / NotificationManager）。

### 5.2 主 cs 接入

`TheSingerOfTheEnd.cs` 的 `OnStarSystemLoaded` 在确认是 `outing_system` 后，`new GameObject("SingerEndingJudge").AddComponent<EndingJudge>()`。该事件每次进入星系（每个循环）触发，判定器随场景重建。

> 检查的是**本循环条件**（非持久），所以结局在"修复扩音器的那个循环"当场触发——这正是设计里的高潮时刻。

# 5.26（二）测试反馈修复 — Ship Log 显示与资产路径

> 上半部分的改动进游戏测试后暴露两个问题，本节记录排查与修复。

## 一、问题现象

1. **飞船日志提示"已更新"，但打开后空无一物**。
2. **星球表面依旧荒凉**——说明 prop 资产路径完全无效，建筑/角色根本没加载出来。

## 二、Ship Log 为空的根因与修复

查 [NH Ship Log 文档](https://nh.outerwildsmods.com/guides/ship-log/) 后定位到两点：

- **`mapMode.manualPosition` 必须与 `manualNavigationPosition` 成对出现**，否则 map 模式直接不渲染。我之前只写了 `manualPosition`，导致 map 模式整页空白。文档原文：*不设 manualPosition 就走自动模式*。
- **rumor（谣言）模式的卡片排布依赖星系配置里的 `entryPositions`**，缺失时卡片堆在默认行、容易跑到屏幕外看不见。

**修复**：

| 文件 | 改动 |
|------|------|
| `singer_world.json` / `god_realm.json` | 删除 `ShipLog.mapMode.manualPosition` → 改用**自动 map 模式**（NH 按轨道自动摆放星球节点）|
| `outing_system.json` | 新增 `entryPositions`，为 8 个 Entry（SINGER_CITY / CITY_SQUARE / CITY_HALL / CITY_TOWER / GOD_REALM / BET_STELE / GOD_VOICE / THIRD_PARTY）指定 rumor 模式坐标 |

> 注：日志"能提示更新"说明 XML 本身是加载成功的（fact 存在才会触发揭示通知），所以这是纯**显示/排布**问题，不是 XML 没加载。

## 三、资产路径全部替换为验证过的基础游戏路径

读取社区维护的 [Props 列表（Google 表格）](https://docs.google.com/spreadsheets/d/1VJaglB1kRL0VqaXhvXepIeymo93zqhWex-j7_QDm6NE/edit?gid=0#gid=0)，逐条核对。结论：

# 5.26（三）场景丰富化 — 启用 DLC 资产，参考原版布景手法

> 进游戏后物品能正常出现，但场景过于简陋。本轮大幅丰富布景。用户已购买 DLC（Echoes of the Eye），因此本轮启用 Stranger/梦境世界资产。

## 一、布景设计思路（借鉴原版"分层布景"）

原版星球之所以耐看，靠的是**分层**：大型建筑（剪影）+ 中型结构（连接）+ 小型道具（叙事）+ 散布植被（铺底）+ 灯光（氛围）。本轮按这个思路重做世末之城：

| 层次 | 用的资产 | 作用 |
|------|---------|------|
| 大型建筑 | Stranger 木屋群（DLC）：两层木屋 / 神器木屋 / 梦火木屋 / 残破木屋 / 焚毁滑梯木屋 / 大型·超大型投影木屋 | 构成一座真正"有房子"的废城 |
| 竖向地标 | 挪麦重力炮 | 城市边缘高塔轮廓 |
| 古迹层 | 挪麦石质废墟 Ruin_Empty / Ruin_Campfire | 比木屋更古老的一层，叙事纵深 |
| 连接结构 | Owlk 桥段、发光挪麦门廊 | 引导动线 |
| 灯光 | 赫斯人蓝灯笼 ×4、挪麦立式火炬 ×2 | 昏暗多雨城市的关键氛围 |
| 道具 | 挪麦长桌 / 长椅 / 书架 | 广场与音乐厅的生活气息 |
| 植被散布 | 梦境湿地树 ×30、低矮灌木 ×12、枯灌木 ×20、陶器碎片 ×25 | 全星球铺底，消除荒凉 |

> **氛围契合**：Stranger（回声）的木质建筑 + 梦境世界的湿地树，本身就是"衰败、潮湿、孤寂"的美学，与《世末歌者》"困于鸥停之雨的废城"高度契合，比纯挪麦石建筑更贴主题。

## 二、关键修正：按球面半径校正每个 prop 的高度

之前 prop 的 `y` 值是随手填的（多为 290/300/301），但星球是半径约 301 的球面——离极点越远，地表的 y 越低。例如塔在 (x=90,z=55) 处地表 y≈282，却被放在 y=300，**悬空约 18 米**。这也是"显得简陋"的一个原因。

本轮对每个 detail 用 `y ≈ √(301² − x² − z²)` 重新计算贴地高度（神谕之境用半径 101 同理）。信号源、对话体、揭示体、占位角色的高度也一并对齐。

> 后续仍建议进游戏用 Debug Raycast（按 P）做最后微调——大型建筑的 pivot 不一定在底部。

## 三、神谕之境同步丰富

# 5.26（四）挪麦文字修复 + 地表材质（HeightMap）

> 进游戏反馈：① 挪麦文字无法翻译、且凭空漂浮（未依附墙面）；② 星球表面是一颗光秃秃的白球，无材质、无起伏，世末之城显得大而空旷。

## 一、挪麦文字：改用 `cairn`(石碑) 类型

查 NH 的 `TranslatorTextInfo` schema 发现两个问题：

1. **`alignRadial` 不是 translatorText 的合法字段**（它只属于 `details`）。我之前给文字写了 `alignRadial`，根本没生效，导致朝向错乱、文字漂浮。translatorText 的朝向应由 **`normal`**（文字面朝的方向）控制。
2. `type:"wall"` 的螺旋文字本身不带墙体几何——它需要贴在一面真实的墙上，否则就是一组悬空的弧线，且不好对准翻译。

**修复**：把歌者日记和赌约石碑都改成 **`type:"cairn"`**——挪麦石碑是**自带石质几何体、立在地面、必定可被翻译器对准**的物件，正好对应"石碑"的设定，从根本上消除漂浮与无法翻译的问题。同时用 `normal`（径向朝上）让石碑立直，去掉了非法的 `alignRadial` / `seed`。

```json
"translatorText": [
    {
        "xmlFile": "planets/text/bet_stele.xml",
        "type": "cairn",
        "position": { "x": 0, "y": 101, "z": 0 },
        "normal": { "x": 0, "y": 1, "z": 0 }
    }
]
```

> 若后续仍想要"墙上的螺旋文字"的原版观感，需要：放一面挪麦白板/墙体 prop，再用 `type:"wall"` + 正确的 `normal`/`parentPath` 贴上去——这一步需要进游戏反复对位，故本轮先用必定可用的 cairn。

## 二、地表材质：HeightMap + 程序化贴图

白球的根因是星球没有 `HeightMap`（地表颜色贴图 + 高度贴图）。本轮用 numpy 程序化生成了等距柱状投影贴图：

| 文件 | 内容 |
|------|------|
| `planets/textures/singer_ground.png` | 世末之城·阴湿青灰废墟色（分形噪声 + 积水暗斑）|
| `planets/textures/singer_height.png` | 世末之城·轻微起伏高度图 |
| `planets/textures/god_ground.png` | 神谕之境·淡紫白结晶色 |
| `planets/textures/god_height.png` | 神谕之境·轻微起伏高度图 |

生成脚本在 `TheSingerOfTheEnd/gen_textures.py`（已移出 `planets/`，不打包进 MOD，可随时改色重跑）。

星球配置加 `HeightMap`：

```json
"HeightMap": {
    "textureMap": "planets/textures/singer_ground.png",
    "heightMap": "planets/textures/singer_height.png",
    "minHeight": 300, "maxHeight": 302, "smoothness": 0.1
}
```

**关键设计**：起伏区间设得很窄（半径 300–302，仅 ±1）。因为之前所有 prop 都是按"到球心距离 ≈ 301"摆放的，把地形也压在 301 附近，**所有建筑/道具就自动贴合新地表，不用再挪一遍**。贴图提供颜色/材质，轻微起伏打破"完美光球"的呆板感。

配套微调：
- `Base.groundSize` 降到 298 / 98（低于地形最低处），作为隐藏的兜底球面，避免与 HeightMap 地表叠加穿模。
- 出生点抬到 y=305 / 飞船 306（高于地形峰值 302），落地不会卡进地形。

## 三、关于"体积大、空旷"

本轮**没有缩小星球**——因为半径一改，几十个 prop 的坐标就得全部重算，风险高。改为用**地表材质 + 起伏 + 浓雾**来解决"空"的观感：世末之城 `fogDensity 0.55`，可视距离本就有限，玩家看不到整颗空球；加上地表纹理和城区密集的建筑，空旷感会明显改善。
若你看过实际效果后仍觉得太大，告诉我，我再做一次"缩小 + 整体重摆"的协调改动。
