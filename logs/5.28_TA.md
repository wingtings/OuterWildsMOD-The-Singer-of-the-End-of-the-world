# 5.28 技术美术 TA 调研
0. 先纠正一个会浪费大量时间的认知误区

▎ Blender 里"连节点"做出来的渲染效果，几乎无法通过 FBX 带进 Unity/OW。

FBX 只携带：网格、UV、法线、骨骼+蒙皮权重、BlendShape（MMD 表情 morph）、材质槽 + 贴图引用。它不携带 Blender 的 Shader 节点图（Principled / Shader-to-RGB / ColorRamp 这些）。导入 Unity 时材质会被重建为 Standard（你现在的 Body.mat 等 30+ 材质就是 m_Shader: fileID 46 = 内置 Standard，所以模型现在看起来发灰发平）。

所以你的需求要拆成两条互不相同的路：

┌────────────────────────────────────────┬──────────────────────┬──────────────────────────────────────────────┐
│                  目标                  │ Blender 节点有没有用 │             真正决定画面的是什么             │
├────────────────────────────────────────┼──────────────────────┼──────────────────────────────────────────────┤
│ A. 在 Blender 里出图/出宣传片/过场视频 │ 有用，核心工作       │ Blender 节点组 + EEVEE                       │
├────────────────────────────────────────┼──────────────────────┼──────────────────────────────────────────────┤
│ B. 让模型在游戏(OW)里好看              │ 没用（导出即丢）     │ Unity 端的卡通 Shader + 干净的网格/法线/权重 │
└────────────────────────────────────────┴──────────────────────┴──────────────────────────────────────────────┘

你做的是游戏 MOD，主战场是 B。Blender 的价值在于：① 清理网格/法线/权重/表情；② 烘焙能带走的贴图（AO、法线）。真正的卡通外观必须在 Unity 里用卡通 shader 重做。下面据此安排。

1. 现状盘点

- 已有 6 个屏幕/材质 shader：GodRay(3-pass 后处理)、VolumetricRain(粒子)、RainRipple(地面水洼)、AudioWave(FFT 环)、VolumetricFog(Ray-march 后处理)、WaterReflection(平面反射)。代码质量不错。

- 彩蛋：模型自带 外套音频可视化/耳机波形/耳机音频可视化/外套心跳/耳机心跳 这些材质 —— 原 PMX 作者就是留给你做自发光 + 律动的，正好接你的 FFT 系统（见 §4）。

3. Blender 渲染方案（详细）

3.1 导入与清理（A、B 两条路都要做）

1. 启用 mmd_tools（File ▸ Import ▸ MikuMikuDance Model (.pmx)）。你材质名里有 mmd_tools_rigid_*，说明你已经走过这步。
2. （可选）Cats Blender Plugin ▸ Fix Model：例（MMD 单位≈0.08m，Cats 会归一到≈1.0）。注意
理（卡通的命脉）：Shade Smooth + 开 Auto Smooth + 加 Weighted Normal 修改器。法线乱 → 卡通色阶和描边一定会破。
4. 权重上限 4：Unity FBX 每顶点最多 4 骨骼，Weights ▸ Limit Total = 4。
5. 确认 Shape Keys（口型/表情 morph） 完整 —— 这是少数能带进 Unity 的"动画资产"。

3.2 成熟的卡通节点方案（仅用于在 Blender 里出图/烘焙，不进游戏）

EEVEE 下业界通用的最简卡通节点链（每个材质）：

图像纹理(基础贴图) ─► 漫射BSDF ─► [Shader to RGB] ─► [ColorRamp(插值=Constant, 2~3档)] ─► 自发光/基础色 ─► 材质输出
                                                         └─(可叠) Fresnel ─► 混合 ► 轮廓光
- Shader to RGB 把连续光照转成可被 ColorRamp 切成"硬色阶"的值 —— 这就是卡通分界。
- mmd_tools 导入时其实已生成一个 mmd_shader 节点组（接好了 base/toon/sphere），可直接基于它改。
- 描边：用 Solidify 修改器（翻转法线/Flip + Material Offset 指到一个纯黑、Backface Culling 的材质） 做反向壳描边；或用 Freestyle 线条渲染出静态图。

3.3 能真正带进 Unity 的 Blender 产出 —— 烘焙

节点本身进不去，但烘焙出来的贴图可以：
- Render ▸ Bake ▸ Ambient Occlusion → AO 贴
- 高模→低模烘 Normal Map（如果你要给低面数版加细节）。
- 这些是普通 PNG，Unity 端正常用。                                                       
3.4 FBX 导出设置（关键）

File ▸ Export ▸ FBX：
- Limit to Selected Objects，Object Types 只勾 Mesh + Armature。
- Transform：Apply Scalings = FBX All，Forward = -Z Forward，Up = Y Up（Unity 约定）。   - Geometry：勾 Apply Modifiers（若想把描边壳）、Custom Normals 一定要带。
入设置（选中 fbx，Inspector）

- Model 页：Scale 核对（角色应≈1.6~1.8m 高）；Normals = Import（保留你在 Blender 调好的自定义法线，别让 Unity 重算把卡通法线毁了）；Blend Shapes 勾选。
- Rig 页：要用 Mecanim/人形动画就 Humanoid，否则 Generic。
- Materials 页：Extract Materials + Extract Textures 到工程，再逐个换 shader。

4.2 把材质换成 UTS2（核心）

导入 UTS2_ShaderOnly 包后，逐材质设：
- Base Map = MMD 基础贴图；1st/2nd Shade Color = 基础色压暗的两档（参考 MMD toon.bmp 的暗部色）。
- BaseColor Step / Feather 调色阶分界软硬。
- Rim Light 打开 —— 阴雨场景的逆光轮廓非常出彩。
- MatCap：把 MMD 的 SPH(乘) / SPA(加) 球面贴图喂给 UTS2 的 MatCap 槽，保住原模型的"金属/水润高光"。
- Outline：模式选 Normal Direction，宽度小、颜色暗，得到 MMD 招牌描边。
- Emissive：外套音频可视化 / 耳机波形 / 心跳 这几个材质开 Emissive。

▎ 强烈建议的技术美术亮点：你的 AudioVisualizerController 已经在 SetFloatArray("_Spectrum", …)。把这些自发光材质也接上 —— 每帧按 FFT 频段 / 心跳节拍 material.SetColor("_EmissionColor", …)，让歌者的衣服/耳机▎ 声波可视化 + 自发光三件事一次性串起来，演

▎ 30+ 材质手动换太累：写个 Editor 脚本批量 material.shader = Shader.Find("UnityChanToonShader/...") 并把 _MainTex→_BaseMap 搬过去。

4.3 打进 AssetBundle 的坑（OW MOD 必看）

- OW 里没有 UTS2，所以 UTS2 shader 必须被打进 bundle：只要 bundle 里的 Material 引用了 UTS2，shader 会被自动纳入。
- 防 变体剥离：把 UTS2 加入 Project Settings ▸ Graphics ▸ Always Included Shaders，并在装了 UTS2 的工程里执行 BuildAssetBundles，否则进游戏会变粉色(shader 丢失)。
- 用最小的 ShaderOnly 版，别打整包，控制 bundle 体积和变体数。
- 进游戏用 F7/F8 验证：粉色=shader 没打进去；纯黑/过亮=色彩空间或光照问题（UTS2 建议 Linear，OW 本身就是 Linear，匹配）。

5. Unity 后处理方案（OW = 内置管线）

你现有 GodRay/Fog 已经走 OnRenderImage Blit，路线是对的。两种扩展方式：                  
路线 A（推荐，零依赖、与现有代码一致）：再写两个 OnRenderImage Blit —— Bloom（亮度阈值提取 → 高斯模糊 → 叠加）和 Color Grading（LUT/曲线调色）。可控、不和 OW 冲突。
路线 B：用 Post-Processing Stack v2（com.unity.postprocessing，支持内置管线）。OW 本体就在PPv2，所以你甚至可以复用游戏现成的 PostProce自己打 PPv2 程序集。⚠️ 这条要在游戏里验证不与 OW 既有 profile 打架，作为进阶选项。

全屏效果挂载顺序（OnRenderImage 按组件挂载顺序自上而下执行，顺序很关键）：
场景渲染
 └─1) 体积雾 (深度)         ← 先做大气透视
 └─2) God Rays              ← 体积光
 └─3) Bloom                 ← 让"自发光卡通 + 神光"发光晕开
 └─4) Color Grading / 调色  ← 阴雨冷调去饱和；True End 暖调
 └─5) Vignette 暗角         ← 收束注意力
 └─6) (可选) 色差/颗粒       ← 氛围
- 这几个 MonoBehaviour 要挂在 OW 的 PlayerCamera 上，且排在 OW 自身后处理之后，避免被覆盖。
- 叙事联动：你已有 TimelineManager 在驱动 God Ray/雨量。把它扩展为一个统一的"末日↔救赎"调调色更冷、雾更浓、暗角更重；True End 瞬间 →  骤降 + God Ray爆发（"雨停、阳光穿云"）。一个时间轴串起所有

---
6. 建议落地顺序

1. UTS2 接角色（§4）—— 画面立刻"从塑料变二次元"，且直接回应你的 MMD 诉求。
2. 自发光材质接 FFT（§4.2 亮点）—— 角色与声波可视化联动。
3. 收尾现有 6 个 shader 的 Material/bundle（§2 P0.5）。
4. 加 Bloom + 调色（§5）—— 让前面的发光/神光成立。
5. TimelineManager 统一调度末日→True End 的后处理演出（§5 末）。
6. 有余力再做全息 shader和湿身/溶解转场。
